# Project Reflection — Bellini Scheduling System
**Author:** Joshua  **Course:** Project 2 — Team T-03  **Date:** 2026-04-18

Working as the lead architect on the Bellini Scheduling System gave me my first end-to-end exposure to building a non-trivial production-style web application as a software engineer rather than just a coder of isolated assignments. The two user stories I personally owned — **US-09 (cross-semester instructor workload summary with PDF and Excel export)** and **US-10 (waitlist-to-capacity imbalance detection across two consecutive semesters)** — both required pushing logic down into the database via PostgreSQL views (`instructor_workload`, `waitlist_alerts`) rather than computing aggregates in the application layer. That single design choice taught me more about separation of concerns than any prior coursework: the React pages became thin presentation, the Next.js API routes became a thin transport, and the "real" business logic lived where the data already was. When I later had to add a `ta_count` and `instructor_name` to the TA-management page, I felt the cost of that decision too — the view didn't expose those fields, so I had to enrich rows in the query layer. Both the win and the friction were instructive.

Agile worked well for us because we kept the slices small. We mapped each of the ten user stories to its own page, its own use case, and its own Interaction Overview Diagram, which made standups and reviews concrete: every task had a visible artifact a teammate could open in the running app. Designing the IODs in **Mermaid** before writing the page also forced me to think about the *flow* of a feature — the branches, the error states, the "no data" cases — instead of jumping straight to implementation. The IOD-10 I added late for the Audit Reports story is a good example of that: writing the diagram first surfaced four distinct audit branches I would otherwise have built ad hoc.

On the tooling side, I leaned heavily on **TypeScript, Supabase (Postgres + RLS + Realtime), Tailwind, jsPDF, and ExcelJS**. The biggest single lesson was the value of strict typing across a server/client boundary: when the database view's column names drifted from what the React component expected, TypeScript happily rendered nothing rather than crashing — a silent failure that the user caught before I did. I now treat any DB-shape-to-component contract as something that must be explicitly mapped and tested, not assumed.

Teamwork mattered in less obvious ways than I expected. With four contributors (Paul, Julia, Torin, and myself) each owning different stories, our shared schema and shared `(app)` route group became the integration points that everyone touched. We avoided merge conflicts by agreeing on a thin `src/lib/queries` layer per story, but I underestimated how much coordination the *visual* parts — globals.css, the side nav, the semester dropdown — would need. A single dark-mode media query I never thought twice about silently broke half the app for users who had OS-level dark mode enabled.

If I had to summarize one takeaway: a software engineer's job is mostly designing the seams between layers, not writing the code inside them. The code is easy; the contracts are what break.
